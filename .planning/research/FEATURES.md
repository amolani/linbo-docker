# Feature Landscape

**Domain:** linbofs64 boot-pipeline transparency — hook system, content auditing, update safety
**Researched:** 2026-03-10
**Scope:** v1.2 milestone — what features are needed to make hooks reliable, linbofs64 content transparent, and LINBO package updates safe

## How Hooks Work in Practice (Current State)

### Hook invocation sequence (from update-linbofs.sh)

```
Step 1:  Read kernel variant from custom_kernel
Step 2:  Validate kernel variant directory
Step 3:  Hash rsync password
Step 4:  Prepare WORKDIR (mktemp)
Step 5:  Create backup (linbofs64.bak)
Step 6:  Extract linbofs64 template (xzcat | cpio -i, CWD = WORKDIR)
Step 7:  Inject kernel modules (tar xf modules.tar.xz)
Step 8:  Inject password hash (etc/linbo_pwhash, etc/linbo_salt)
Step 9:  Inject SSH keys (etc/dropbear/, etc/ssh/, .ssh/authorized_keys)
Step 10: Copy default start.conf
Step 10.5: Inject firmware files (lib/firmware/)
Step 10.6: Inject wpa_supplicant.conf
Step 10.7: Inject GUI themes + custom linbo_gui binary
Step 10.9: *** RUN PRE-HOOKS (exec_hooks pre) ***  <- CWD = WORKDIR (extracted linbofs root)
Step 11: Repack (find . | cpio | xz + append devnodes CPIO segment)
Step 12: Verify new file (size sanity check >=10MB)
Step 13: Replace original (mv .new -> linbofs64)
Step 14: MD5 hash
Step 14.5: Write .linbofs-patch-status marker
Step 14.6: Sync to Docker volume (if LINBO_DIR != /var/lib/docker/volumes/...)
Step 15: Copy kernel binary from variant
Step 15.5: *** RUN POST-HOOKS (exec_hooks post) ***  <- CWD = WORKDIR (but CPIO already done)
```

### What hooks currently receive

Exported variables available in both pre and post hooks:

| Variable | Value | Notes |
|----------|-------|-------|
| `LINBO_DIR` | `/srv/linbo` | Boot files, images |
| `CONFIG_DIR` | `/etc/linuxmuster/linbo` | SSH keys, rsyncd.secrets, firmware config |
| `CACHE_DIR` | `/var/cache/linbo` | Build cache, mktemp location |
| `KTYPE` | `stable` / `longterm` / `legacy` | Active kernel variant |
| `KVERS` | `6.18.4` (or empty if no variant) | Kernel version string |
| `WORKDIR` | `/var/cache/linbo/linbofs-build.XXXXXX` | Extracted initrd root (mktemp dir) |

### Pre-hook working directory

Pre-hooks run with `CWD = WORKDIR`, which is the extracted cpio root. Relative paths like `usr/share/plymouth/themes/` work directly. This is intentional and mirrors the LMN upstream pattern.

### Post-hook working directory

Post-hooks also run with `CWD = WORKDIR`, but the directory is empty-ish at this point — the cpio archive is already assembled and written. The final `linbofs64` file is at `$LINBO_DIR/linbofs64`. Post-hooks are for notifications, side effects, and deployment triggers, not for content modification.

### Active hook: 01_edulution-plymouth

Location: `/etc/linuxmuster/linbo/hooks/update-linbofs.pre.d/01_edulution-plymouth`
Effect: Replaces `usr/share/plymouth/themes/linbo-splash/linbo-splash.script` and `logo_no_arrows.png`, removes `arrows-*.png`.
Failure mode: If `THEME_SRC` directory missing, silently exits 0 (safe degradation).

### Hook error behavior

Hooks that exit non-zero generate `WARNING: hook foo exited with N` but do NOT abort the build. This is intentional — a broken branding hook should not prevent boot files from being updated.

## What Is Inside linbofs64 (Docker vs LMN)

### Docker linbofs64 composition

The Docker linbofs64 is assembled from:

1. **Base template:** `$KERNEL_VAR_DIR/linbofs64.xz` — vanilla LINBO initrd from the linuxmuster-linbo7 deb package, extracted into the kernel set by `linbo-update.service.js:provisionKernels()`. This is identical to what LMN's own `update-linbofs` uses as its input.

2. **Kernel modules:** `$KERNEL_VAR_DIR/$KTYPE/modules.tar.xz` — module set built for the selected kernel variant (stable/longterm/legacy). The Docker module set is built by the init container from the same deb package. This is where Docker and LMN diverge: the LMN `update-linbofs` builds its module set from the modules installed on the host via the deb, while Docker uses the tarball that was pre-packaged in the deb's `/var/lib/linuxmuster/linbo/` directory.

3. **Password hash:** `etc/linbo_pwhash` + `etc/linbo_salt` — argon2 hash of the rsync linbo password.

4. **SSH keys:** `etc/dropbear/*_host_key`, `etc/ssh/ssh_host_*_key*`, `.ssh/authorized_keys` — auto-generated by init container, persisted in CONFIG_DIR volume.

5. **Firmware files:** `lib/firmware/` — populated from `$CONFIG_DIR/firmware` manifest if present.

6. **wpa_supplicant.conf:** `etc/wpa_supplicant.conf` — injected if present in CONFIG_DIR.

7. **GUI themes:** `themes/$name/` — from `$LINBO_DIR/gui-themes/`.

8. **Default start.conf:** Copied from `$LINBO_DIR/start.conf` if present.

9. **Pre-hook modifications:** Any files modified/added by pre-hooks (currently: Plymouth theme replacement).

### Key difference: Docker vs vanilla LMN

The critical known divergence (Session 32 root cause): Docker's `modules.tar.xz` for a given kernel version contains a DIFFERENT module selection than what LMN's `update-linbofs` produces, even when both use the same kernel version. LMN builds modules from the live kernel on the server, selecting all modules under `/lib/modules/$(uname -r)/`. The Docker init container pre-packages a specific subset. Symptom: "Remote Control Mode" on clients that need modules outside the Docker subset.

### init.sh DHCP serverid problem

The init.sh (inside linbofs64) `do_env()` function overwrites the `server=` cmdline parameter with the DHCP `serverid` when `HOSTGROUP` is set. In Docker deployments where the DHCP server (10.0.0.11) is different from the LINBO server (10.0.0.13), the client rsync target becomes wrong. This is the only known case where vanilla init.sh behavior is actively harmful to Docker deployments.

## Table Stakes

Features users expect for this milestone. Missing = hooks unreliable, content opaque, updates risky.

| Feature | Why Expected | Complexity | Notes |
|---------|--------------|------------|-------|
| **Hook content manifest** | After every build, an admin needs to know what was actually injected. The `.linbofs-patch-status` only records "OK". A manifest listing which hooks ran, what files changed, module count, key count, firmware count makes the build auditable. | Low | Extend Step 14.5. Can be shell-only. |
| **Hook listing/status API** | The web UI should show which hooks are installed, whether they ran last build, and their exit codes. Currently there is no API surface for hooks at all. Admins discovering hook failures only see WARNING in build output. | Medium | New GET /system/hooks endpoint in linbofs sub-router. Reads HOOKSDIR, returns list with name, executable status, last_exit_code from build log. |
| **linbofs content audit command** | A shell command (or make target) that extracts linbofs64 and reports its composition: kernel version in lib/modules/, SSH keys present, password hash present, firmware files, hook-injected files. Allows before/after comparison. | Low | `make linbofs-audit` — shell one-liner wrapping xz+cpio+grep. |
| **Hook validation on install** | When a hook is added to the pre/post directory, validate it before the next build: is it executable, has it a shebang, does it reference valid paths? Catching broken hooks at install time is better than catching them during a live build. | Low | Shell helper `validate-hook.sh` or inline in a new `make hooks` target. |
| **init.sh SERVERID patch as pre-hook** | The `do_env()` DHCP serverid overwrite is the only confirmed vanilla init.sh behavior that breaks Docker deployments. It should be patched as a pre-hook rather than a direct modification. This is architecturally consistent with the "vanilla LINBO, hooks only" constraint. | Medium | Pre-hook that patches `etc/linuxmuster/linbo/init.sh` inside the extracted WORKDIR. Idempotent (check if patch already applied). Conditional on `LINBO_SERVER_IP` env var being set. |
| **Update safety: hook re-run after linbo7 upgrade** | When `linbo-update.service.js` installs a new linbo7 deb (Step: rebuild linbofs), it calls `linbofsService.updateLinbofs()` which runs update-linbofs.sh, which runs hooks. This is correct. But it needs to be documented and tested: hooks MUST survive a package update. | Low | Existing flow is correct; needs test coverage and documentation. |
| **Build log retention** | The output of update-linbofs.sh (including hook stdout) is currently captured by linbofsService.updateLinbofs() into `result.output` but is not persisted anywhere. Admins have no way to see what the last build did without triggering a new build. | Medium | Write build output to `$LINBO_DIR/.linbofs-build.log` in update-linbofs.sh. Rotate (keep last 3). API endpoint to retrieve it. |

## Differentiators

Features that would add significant value beyond the minimum.

| Feature | Value Proposition | Complexity | Notes |
|---------|-------------------|------------|-------|
| **linbofs diff: Docker vs vanilla** | Show which files Docker's linbofs64 adds/modifies vs the vanilla template. This is the answer to "what does Docker actually change?" and enables regression detection after package updates. | Medium | Shell script: extract both the template (linbofs64.xz from kernel set) and the built linbofs64, diff their cpio manifests. No binary diff needed — just file list + size comparison. |
| **Hook scaffold generator** | `make new-hook NAME=02_my-patch TYPE=pre` creates a hook skeleton with the correct shebang, exported variable documentation, and error handling pattern. Reduces hook authoring friction. | Low | Shell template + make target. |
| **Module diff: Docker vs LMN** | Compare `lib/modules/` contents of Docker linbofs64 vs an LMN-generated linbofs64. This directly addresses the Session 32 root cause. Requires access to an LMN linbofs64 for comparison. | Medium | Shell script that extracts both cpio archives and diffs the module lists. Output: modules in LMN but not Docker, modules in Docker but not LMN. |
| **Firmware audit: what hardware is covered** | Parse `$CONFIG_DIR/firmware` manifest and cross-reference against `linux-firmware` package contents. Report: entries that matched, entries that were not found, total firmware blob count and size. | Medium | Shell or Node.js script. Useful for debugging "NIC not working in LINBO" issues. |
| **Hook idempotency testing** | Run pre-hooks twice on the same extracted linbofs and verify the result is identical. Idempotent hooks are safe to re-run; non-idempotent hooks can cause subtle corruption. | Medium | Test harness in test suite. Relevant for the init.sh serverid patch hook especially. |
| **init.sh serverid patch: configurable guard** | The SERVERID patch hook needs a guard: it should only activate when `LINBO_DOCKER_MODE=true` (or similar env var) to prevent accidental application on pure LMN servers that happen to have the same hook directory. | Low | Env var check at top of hook script. Already available via exported LINBO_DIR path inspection as a proxy. |

## Anti-Features

Features to explicitly NOT build in v1.2.

| Anti-Feature | Why Avoid | What to Do Instead |
|--------------|-----------|-------------------|
| **Modifying init.sh directly** | Violates the core "vanilla LINBO, hooks only" constraint established in Session 30. Every direct modification to init.sh becomes a diff that must be maintained across every linbo7 package update. | Implement the SERVERID fix as a pre-hook that patches init.sh inside WORKDIR only — the live init.sh in the template is never touched. |
| **Hook execution sandboxing** | Running hooks in a chroot or container-within-container adds enormous complexity. Hooks need access to CONFIG_DIR, LINBO_DIR, and WORKDIR. | Accept that hooks run as root with full host access. Document this. Use naming conventions and review processes instead of technical sandboxing. |
| **Auto-generated hooks from UI** | A web UI for writing hook scripts would encourage complex hooks that are hard to debug. Hooks belong in version control, not in a database. | Provide scaffold generator (CLI only). Hooks are shell scripts that admins write and place in the hooks directory. |
| **Hook version locking** | Tracking which hook version was used for a given linbofs64 build adds complexity with little payoff for single-site deployments. | Include hook names and mtimes in the build manifest. That is enough for forensics without a full versioning system. |
| **Binary diff of linbofs64 vs template** | Binary diffs of cpio+xz archives are meaningless to an admin. What matters is the file list and key metadata. | File list diff (cpio -t output comparison) gives 95% of the diagnostic value with 5% of the complexity. |

## MVP Recommendation

The v1.2 "boot pipeline transparency" milestone has three distinct concerns. Each maps to one phase.

### Phase 1: Hook reliability + manifest

**Goal:** Make hooks visible and auditable.

1. **Hook content manifest** — extend Step 14.5 to write a structured manifest (JSON) including hook names, exit codes, file counts per injection step, build timestamp. Write to `$LINBO_DIR/.linbofs-build-manifest.json`.
2. **Build log retention** — write full update-linbofs.sh stdout to `$LINBO_DIR/.linbofs-build.log`, rotated, accessible via API.
3. **Hook listing/status API** — `GET /system/hooks` returns installed hooks, last exit code from manifest, executable status.
4. **Hook validation** — add `validate-hook.sh` script that checks shebang, executable bit, and that any referenced paths are absolute.

Confidence: All of this is additive to existing code. No architectural changes.

### Phase 2: linbofs content audit

**Goal:** Answer "what is actually inside the linbofs64 that clients download?"

5. **linbofs content audit command** — `make linbofs-audit` that produces: kernel version, module count, SSH key fingerprints, firmware files, hook-modified files vs template baseline.
6. **linbofs diff: Docker vs vanilla** — script that compares `linbofs64.xz` (template) vs `linbofs64` (built), reports additions/modifications.
7. **Module diff** — specialized script to compare lib/modules/ contents between Docker and LMN linbofs64 (requires LMN linbofs64 as input file, optional feature).

Confidence: All shell-based. No Node.js changes needed.

### Phase 3: init.sh SERVERID patch + update safety

**Goal:** Fix the only known vanilla init.sh behavior that breaks Docker deployments, and verify hooks survive package updates.

8. **init.sh SERVERID patch as pre-hook** — implement `02_docker-init-serverid` pre-hook that patches `init.sh` inside WORKDIR to preserve `server=` cmdline parameter over DHCP serverid. Idempotent, guarded by `LINBO_DOCKER_MODE=true`.
9. **Update safety test** — integration test that: installs a new linbo7 package simulation, runs update-linbofs, verifies all hooks ran, verifies SERVERID patch is present in output linbofs64.
10. **Hook scaffold generator** — `make new-hook` CLI target.

Confidence: The SERVERID patch approach is MEDIUM confidence — requires verifying the exact patch location in the current init.sh version and that patching-inside-WORKDIR is reliably idempotent (the patch must detect whether it was already applied).

## init.sh SERVERID Patch: Feasibility as a Hook

### What the patch must do

In `init.sh` function `do_env()`, the line that overwrites the server with DHCP serverid looks like:

```sh
server=$(grep -i "^serverid" $DHCP_FILE | head -1 | awk '{print $2}')
```

or similar (exact form depends on init.sh version). The patch must conditionally skip this assignment when a `server=` parameter was present on the kernel cmdline.

### Hook implementation approach

```
pre-hook: 02_docker-init-serverid
  1. Check LINBO_DOCKER_MODE=true (guard)
  2. Locate init.sh in WORKDIR (should be at ./init.sh or ./linuxmuster/linbo/init.sh)
  3. Check if patch already applied (idempotency marker comment)
  4. Apply sed patch: wrap the serverid assignment in a conditional
  5. Verify init.sh is still valid shell (bash -n ./init.sh)
```

### Risks

- **init.sh location in cpio** — must verify exact path in the extracted WORKDIR. The vanilla linbofs64 cpio structure has init.sh at the initrd root (`./init.sh`). Confirmed by existing TROUBLESHOOTING.md `cpio -i --to-stdout init.sh` command.
- **init.sh changes between package versions** — the sed pattern must match the correct line. If linbo7 4.4.x restructures do_env(), the patch silently fails. Mitigation: verify patch applied after sed, fail loudly if not.
- **Multi-DHCP-server environments** — if there are cases where using the DHCP serverid is correct, the guard must be opt-in (`LINBO_DOCKER_MODE=true`). Default: patch does NOT apply.

### Confidence: MEDIUM

The patching approach is sound and consistent with "vanilla LINBO, hooks only." The risk is the fragility of text patching across init.sh versions. This risk is manageable by adding a post-patch verification step.
