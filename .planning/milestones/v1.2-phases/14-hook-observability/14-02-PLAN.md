**4. Update getPatchStatus() in linbofs.service.js to parse hook warnings:**

The current `getPatchStatus()` only extracts `buildDate` from .linbofs-patch-status. After Plan 01, the file also contains a `hooks|` line (e.g., `hooks|1 run, 0 warnings` or `hooks|2 run, 1 warnings: 02_custom(exit=1)`).

Update getPatchStatus() to also parse this line:
- Add a regex match for the `hooks|` line: `content.match(/^hooks\|(.+)$/m)`
- Parse the matched value to extract hook count and warning count:
  - `hooks|none` -> `hookWarnings: 0, hookCount: 0`
  - `hooks|N run, M warnings` -> `hookWarnings: M, hookCount: N`
  - `hooks|N run, M warnings: details` -> `hookWarnings: M, hookCount: N, hookWarningDetails: "details"`
- Add these fields to the returned object:
  ```javascript
  return {
    available: true,
    buildDate: dateMatch?.[1]?.trim() || null,
    healthy: true,
    hookWarnings: parsedWarnings,   // number (0 if no hooks line found)
    hookCount: parsedCount,          // number (0 if no hooks line found)
    hookWarningDetails: details,     // string or null
  };
  ```
- Ensure backward compatibility: if the `hooks|` line is absent (older .linbofs-patch-status files), default hookWarnings/hookCount to 0 and hookWarningDetails to null.

This ensures the existing `/system/linbofs-status` endpoint (which calls getPatchStatus()) automatically surfaces hook warning data without any route changes.
